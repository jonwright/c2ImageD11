# docs/AGENTS.md — Living Documentation System

## Core principle

**No human or LLM ever writes documentation content into a `.md` file.**

All API documentation is generated by `tools/generate_docs.py` from:
- `c2ImageD11` module introspection (`dir()`, `help()`)
- `C2PY_BEGIN` blocks in C source files (`"doc"`, `"params"`)
- Function-specific `example.py` / `bench.py` scripts (future)

Generated `.md` files are **never committed to the repo**. They exist only
in CI during doc build, and the rendered site is deployed to GitHub Pages.

## Guardrails (how NOT to hallucinate)

### 1. No generated content in the repo

If you find yourself editing a `.md` file and it contains a function
signature, a help-text block, a parameter table, or a benchmark number:
**stop**. You are doing it wrong. That content comes from
`tools/generate_docs.py`.

### 2. Only hand-written files are hand-written

The following files are hand-written (no auto-generation):

```
docs/index.md                    — project overview (conceptual, no API data)
docs/AGENTS.md                   — this file
docs/guide/variants.md           — ISA dispatch concepts
docs/guide/compiler.md           — compiler drag race concepts
```

Everything else under `docs/api/` is **generated at build time** and must
never be committed.

### 3. How to add a usage example

DO: write an `example.py` (or `examples.py`) in the function's directory:

```python
# lib/functions/score/examples.py
def example_score_basic():
    """Return (code_str, output_str) for doc rendering."""
    import numpy as np, c2ImageD11 as ci
    ubi = np.eye(3)
    gv = np.random.rand(100, 3)
    result = ci.score(ubi, gv, 0.05)
    return (
        "ubi = np.eye(3)\n"
        "gv = np.random.rand(100, 3)\n"
        "ci.score(ubi, gv, 0.05)\n",
        str(result),
    )
```

Then add a test in `tests/` that imports and validates this function:

```python
def test_score_example():
    from lib.functions.score.examples import example_score_basic
    code_str, output_str = example_score_basic()
    # Run the code string and verify it matches the output string
    ...
```

The `generate_docs.py` script imports the example function, calls it,
and writes both the code and output into fenced markdown. If the example
changes, the docs change. If the example breaks, CI catches it.

DO NOT: invent Python code and paste it into a `.md` file.

### 4. How to add benchmark data

DO: run `bench.py --json` on a real machine and commit the output:

```bash
# On a dedicated performance-testing machine:
python lib/functions/score/bench.py --json > docs/bench/score.json
git add docs/bench/score.json
git commit -m "bench: update score performance data"
```

The `generate_docs.py` script reads `docs/bench/<name>.json` and formats
it as a markdown table. Benchmarks are **never** run during CI doc builds
— CI runners are shared, noisy, and useless for performance numbers.

DO NOT: type "M gv/s: 1152" into a markdown file.
DO NOT: include benchmark execution in `generate_docs.py` or `docs.yml`.

### 5. How to update a function's parameter docs

Edit the `C2PY_BEGIN` block in the C source file:

```c
/* C2PY_BEGIN
 * {
 *     "doc": "Count g-vectors indexed by ubi matrix within tol.",
 *     "params": {
 *         "ubi": "Orientation matrix UBI (9-element flattened double).",
 *         "gv": "G-vectors array, shape (ng, 3) flattened.",
 *         "tol": "Tolerance on |h - round(h)|.",
 *     },
 *     ...
 * }
 C2PY_END */
```

This is the **single source of truth**. The `generate_docs.py` script
reads it from the `.c` file and populates the API reference. The
`harvester.py` script reads it for interface generation. There is no
other copy.

DO NOT: edit a parameter description in any `.md` file.

### 6. What happens in CI

**PR workflow** (`test.yml` → `verify_docs` step):
- Runs `tools/verify_docs.py`
- Checks: all C2PY_BEGIN blocks parse correctly, `help()` works for every
  exported function, generator runs without crash, any example/bench scripts
  have corresponding tests
- Fails on ANY issue — no stale docs, no broken examples

**Docs deploy workflow** (`docs.yml`, on push to `master`):
- Installs c2ImageD11 + ImageD11 (tagged release)
- Runs `tools/generate_docs.py`
- Builds with `mkdocs build`
- Deploys to GitHub Pages

### 7. What to do if you need to change doc content

| You want to... | Edit this file | Then run |
|---|---|---|
| Change a function description | `lib/functions/<name>/<name>.c` (C2PY_BEGIN `"doc"`) | `tools/harvester.py`, `tools/generate_docs.py` |
| Change a parameter description | `lib/functions/<name>/<name>.c` (C2PY_BEGIN `"params"`) | `tools/harvester.py`, `tools/generate_docs.py` |
| Add a usage example | `lib/functions/<name>/examples.py` | Add test, run `tools/generate_docs.py` |
| Add/update benchmark data | `lib/functions/<name>/bench.py` (run `--json`, commit to `docs/bench/`) | `tools/generate_docs.py` |
| Change ISA variant descriptions | `lib/functions/<name>/<variant>.c` (C2PY_BEGIN `"doc"`) | `tools/harvester.py`, `tools/generate_docs.py` |
| Update the project overview | `docs/index.md` | None (hand-written) |
| Update the variants guide | `docs/guide/variants.md` | None (hand-written) |

### 8. Detection of spurious edits

There is no git hook to detect bad edits to generated files — because
generated files are never committed. The guardrail is structural: if a
file doesn't exist in the repo, you can't edit it.

For hand-written files (`docs/index.md`, `docs/guide/*.md`):
- These contain conceptual explanations, not API data
- If an LLM adds a function signature or benchmark number to these files,
  the PR reviewer should reject it
- The `verify_docs.py` script checks that these files do NOT contain
  generated-looking content (e.g., no fenced blocks with function
  signatures — that content belongs in `docs/api/`)

### 9. ImageD11 comparison

The ImageD11 comparison section in the docs is generated by extracting
`help()` output from ImageD11's `_cImageD11` module during doc build.
It is a **separate section** — not mixed into the per-function API
reference. c2ImageD11 functions are presented as the primary API;
ImageD11 equivalents are shown as a cross-reference table for users
migrating from the old f2py module.

The comparison is generated, not hand-written. If ImageD11 adds or
removes functions, the table updates automatically.

### 10. Summary for LLM agents

```
RULE 0: NEVER write API data into a markdown file.
RULE 1: Edit C2PY_BEGIN for function docs, example.py for examples, bench.py for benchmarks.
RULE 2: If it looks like a function signature, it came from help().
RULE 3: If it looks like a benchmark number, it came from bench.py --md.
RULE 4: If it looks like a parameter table, it came from C2PY_BEGIN.
RULE 5: If it looks hand-written, it's in docs/index.md or docs/guide/*.md.
RULE 6: When in doubt, run tools/verify_docs.py.
```
