# c2ImageD11 Status — June 22, 2026

## New branch: `new_build_system`

This branch implements the two-tier build architecture (libc2ImageD11.a
via meson + Python extension via setuptools) with yaml embedded in C
source files as `/* @c2py ... */` comment blocks.

## What we proved on `build_system_explore`

The exploration branch (commits on `build_system_explore`) tested:

| Question | Answer |
|----------|--------|
| Does c2py23 handle shape-based dispatch? | Yes — `when:` on `shape[0]==3` / `shape[1]==3` |
| Does it handle stride-based, C+F contiguous? | Yes — `fast_axis`/`slow_axis`/`slow_dim`/`fast_dim` |
| Does `default:false` work? | Yes — floor variants skipped by auto-resolve |
| Does `_variants_*()` work? | Yes — enumerates all variants |
| Does `nearbyint()` outperform magic-constant? | Yes — 3-6× faster, vectorizes, safe with `-ffast-math` |
| Does f32 give 2× at scale? | Yes — halved memory bandwidth at 78M peaks |
| Works on both GCC and clang? | Yes — both give ~1 ns/peak with nearbyint |
| Are variant names = C function names? | Yes — c2py23 v0.2.0 enforces this |
| What build flags are essential? | `-msse4.2` for baseline (ROUNDPD), `-mavx2`, `-mavx512f` |
| Operator precedence in `when:` | `A or B and C` → `A or (B and C)`; need `(A or B) and C` |

## Plan: `new_build_system`

Create the two-tier directory structure.  Do not move C source files
— `meson.build` references them by relative path from `lib/`.

```
lib/                        # Tier 1: meson static library
  meson.build               # builds libc2ImageD11.a from ../src/*.c
  include/                  # public header
    c2ImageD11_api.h        # combined extern "C" API surface

interface/
  manifest.c2py             # module-level settings, file list for harvester
  _cImageD11.c2py           # assembled by harvester (gitignored)
  _cImageD11_bslz4.c2py    # existing generated file (stays)

tools/
  yaml_harvester.py         # NEW: extracts @c2py blocks, assembles .c2py
  generate_bslz4.py         # existing
  extract_kernels.py        # existing
  verify_identity.py        # NEW: anti-hallucination tool (see below)

```

### Phase 2 — YAML harvester (`tools/yaml_harvester.py`)

Inputs:
- `interface/manifest.c2py` — module settings + ordered list of source files
- `src/*.c` — source files with `/* @c2py ... */` comment blocks

Output:
- `interface/_cImageD11.c2py` — complete assembled interface file

Operation:
1. Parse `manifest.c2py` for module name, settings, file list
2. For each source file in the list, scan for `/* @c2py ... */` blocks
3. Extract the yaml content (strip leading ` * ` from each line)
4. Parse with `yaml.safe_load()`
5. Group by function name, merge variant fragments into their groups
6. Assemble into final `.c2py` output

Manifest format:
```yaml
module: _cImageD11
timing: true
free_threading: true

sources:                          # ordered list; yaml fragments extracted in this order
  - src/core/cimaged11utils.c
  - src/geometry/closest.c
  - ...

headers:                          # passed through to generated wrapper
  - src/core/cImageD11.h
  - ...
  - c2py_amd64.h

include:                          # pre-generated c2py fragments appended verbatim
  - interface/_cImageD11_bslz4.c2py
```

### Phase 3 — `@c2py` comment block convention

Three block types, all inside `/* @c2py ... */` comments:

**Type A: Python function definition** (on the reference C function)
```c
/* @c2py function
 * py_sig: "score(ubi: buffer, gv: buffer, tol: float) -> int"
 * doc: "Count g-vectors indexed by ubi matrix within tol."
 * checks:
 *   - "ubi.ndim == 2" / "ubi.shape[0] == 3" / ...
 * gil_release: true
 * c_overloads:
 *   - when: "..."
 *     group: score_aos_f64
 *     map: {...}
 * default_raise: "ValueError: ..."
 */
```

**Type B: Variant marker** (on each SIMD kernel function)
```c
/* @c2py variant
 * group: score_aos_f64
 * when: "c2py_amd64_avx512f"
 * default: true
 */
int score_aos_f64_fast_avx512(...) { ... }
```

**Type C: Module settings** (in `manifest.c2py` only — not embedded in C)

The harvester merges:
- Type A provides the py_sig, checks, c_overloads with group/when/map
- Type B provides the variant C function names and their when: conditions

### Phase 4 — Move yaml into C files (58 functions)

For each of the 58 Python signatures currently in
`interface/_cImageD11_base.c2py`:

1. Find the C source file defining the reference implementation
2. Add a `/* @c2py function */` block above it
3. If the function has SIMD variants, add `/* @c2py variant */` blocks
   above each SIMD kernel function
4. Remove the corresponding section from `interface/_cImageD11_base.c2py`
5. Verify the harvester assembles the same output

Apply c2py23 v0.2.0 conventions while moving:
- Remove explicit `name:` from variants (C function name = variant name)
- Add `default: false` on floor/baseline variants
- Use `fast_axis`/`slow_axis`/`slow_dim`/`fast_dim` in when: conditions
- Remove `buf.contiguous` references (c2py23 removed it)
- Add proper `(A or B) and C` parenthesization to when: conditions
- Add `gv.format` and `ubi.format` checks to every when:

### Phase 5 — Build system

**lib/meson.build** (Tier 1):
- Compiles all C sources from `../src/`
- Multi-flag compilation for SIMD kernels (same as current setup.py)
- Produces `libc2ImageD11.a`

**setup.py** (Tier 2):
- Calls yaml_harvester.py as a build step (or relies on pre-generated .c2py)
- Finds `libc2ImageD11.a` from meson build
- Links it via `extra_objects`
- Only compiles: c2py23 wrapper, c2py_runtime.c, _wrappers.c
- No SIMD compilation, no ISA flags, no subprocess compiler calls

## Anti-Hallucination Strategy for LLM-Assisted Refactoring

### Problem

During the yaml-on-C-sources migration, LLMs may:
- Rewrite function bodies instead of just moving yaml blocks
- Change variable names, types, or logic unintentionally
- Drop or duplicate code blocks
- Change build configuration silently

### Solution: Before/After Identity Verification

**`tools/verify_identity.py`** — strips comments and normalizes C code to
canonical form, then compares the pre- and post-refactoring outputs.

Pipeline:
```
OLD: C source + .c2py interface     NEW: C source with @c2py + harvester output
     |                                      |
     v                                      v
  clang-tidy --fix --checks='-*,readability-*'   same
     |                                      |
     v                                      v
  canonical form (strip comments, normalize)   canonical form
     |                                      |
     v                                      v
  gcc -E -P -C                              same
     |                                      |
     v                                      v
  diff                                      │
     |                                      │
     v                                      v
  MUST SHOW: yaml content moved to comments  EXECUTABLE code:
  EXECUTABLE code must be IDENTICAL         identical diff output
```

Write an AGENTS.md reference script for an LLM to call before every commit.

### Additional Guardrails

1. **Comment-only diffs**: After the migration, `git diff` against the
   reference should show only added comments (the `@c2py` blocks) and
   removed yaml lines.  Any change to a non-comment line is a red flag.

2. **Equivalence tests**: Run `tests/test_equivalence.py` against
   ImageD11._cImageD11 after every batch of function migrations.
   53/54 must pass (TestRefineAssigned skipped).

3. **Buffer tests**: Run `tests/test_buffer.py` — verifies the build
   succeeds and buffer protocol calls work.

4. **Function-by-function commits**: Move one function's yaml at a time,
   committing after each with a message naming the function.  Makes
   bisection trivial if a break occurs.

5. **Preprocessor output comparison**: `gcc -E -P -C src/foo.c | diff -`
   before and after the change.  Only comments should differ.

6. **Checkin guard**: A pre-commit hook runs `verify_identity.py` on
   all changed `.c` files, comparing preprocessed output against the
   pre-refactoring baseline.

### `verify_identity.py` specification

Takes two paths: `old_dir` (pre-refactoring with external .c2py) and
`new_dir` (post-refactoring with @c2py comments). For each `.c` file
that changed:

1. Run `gcc -fpreprocessed -dD -E -P file.c` → `canonical.c`
   (This removes comments, expands macros, normalizes whitespace)
2. Diff the canonical outputs
3. If diff is non-empty, flag as POTENTIAL HALLUCINATION

The output tells the LLM: "these lines changed in executable code, were
you supposed to change them?"
